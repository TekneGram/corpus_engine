# Binary Corpus Architecture with Reverse Indexing and Dependency Parsing

## 1. What Reverse Indexing Is

Reverse indexing (inverted indexing) stores a mapping from an attribute
value to the list of token positions where that value occurs. The
primary corpus storage is position-oriented:

    position → word
    position → lemma
    position → POS
    position → head
    position → deprel

This answers the question: *What is at token position i?*

A reverse index flips that mapping:

    lemma → [positions]
    POS → [positions]
    head → [child positions]
    bundle → [start positions]

Instead of scanning the entire corpus to find where a lemma appears, the
system directly retrieves the sorted list of positions from the index.

### Performance Property

If the corpus has N tokens:

-   Each token contributes exactly one position to the lemma index
-   Each token contributes exactly one position to the POS index
-   Each token contributes exactly one child entry in the reverse
    dependency index

So total storage per index = N integers.

Search complexity becomes:

-   Frequency lookup → O(1)
-   Retrieve occurrences → O(k)
-   Pattern matching → O(min(k₁, k₂))

Where k is frequency of the queried item.

## 2. Binary Architecture with UDParse, N-grams, and Structural Boundaries

The corpus is stored as one contiguous global token stream.

### Core Token Arrays (Parallel Int32 Arrays)

Each file length = N tokens.

    word.bin        → word_id per token
    lemma.bin       → lemma_id per token
    pos.bin         → pos_id per token
    head.bin        → global head position per token (or -1 for root)
    deprel.bin      → deprel_id per token

All arrays align by token index:

    Token position i:
        word[i]
        lemma[i]
        pos[i]
        head[i]
        deprel[i]

### Structural Files

    sentence_bounds.bin   → [start_token_positions]
    doc_ranges.bin        → document_id → (token_start, token_end)
    word_doc.bin          → doc_id per token position

Subcorpus membership is handled in SQLite:

    document table:
        document_id
        title
        author
        subcorpus_id

The binary corpus remains unchanged when metadata changes.

## Inverted Index Files

These posting-oriented artifacts may be emitted in raw binary form or
in compressed form. Compressed postings use delta + variable-byte
encoding.

### Lemma Index

    lemma.index.header
        lemma_id → (offset, length)

    lemma.index.positions
        flattened sorted token positions

Total integers stored = N.

### POS Index

Same structure as lemma index.

### Word Index

Same structure as lemma index.

### Reverse Dependency Index

Built from head.bin:

    dep.index.header
        head_position → (offset, length)

    dep.index.positions
        flattened child positions

Each token appears exactly once as a child (except roots).

Total integers stored ≈ N.

## 2-grams, 3-grams and 4-grams

Generated by sliding window over word.bin without crossing sentence
boundaries. These bundle IDs are word-based, not lemma-based.

For a sentence:

    [word0, word1, word2, word3, word4]

2-grams:

    (0,1)
    (1,2)
    (2,3)
    (3,4)

3-grams:

    (0,1,2)
    (1,2,3)
    (2,3,4)

4-grams:

    (0,1,2,3)
    (1,2,3,4)

### Bundle Storage

    2gram.lexicon.bin
        2gram_id → (word1, word2)

    2gram.freq.bin
        2gram_id → global frequency

    2gram.index.header (optional)
    2gram.index.positions (optional)

    3gram.lexicon.bin
        3gram_id → (word1, word2, word3)

    3gram.freq.bin
        3gram_id → global frequency

    3gram.index.header (optional)
    3gram.index.positions (optional)

    4gram.lexicon.bin
        4gram_id → (word1, word2, word3, word4)

    4gram.freq.bin
        4gram_id → global frequency

    4gram.index.header (optional)
    4gram.index.positions (optional)

Same structure for all n-gram families. `4gram` position indexes may be
pruned to bundles with frequency ≥ 2.

Each feature family may also write document-frequency postings:

    word.docfreq.*
    lemma.docfreq.*
    2gram.docfreq.*
    3gram.docfreq.*
    4gram.docfreq.*

Total stored positions across all bundles ≈ N.

## 4. Why Reverse Dependency Index Is Helpful

UDParse stores dependencies as:

    child_position → head_position

Without reverse index:

To find dependents of a verb at position p, you must scan all tokens:

    for i in 0..N:
        if head[i] == p:
            match

Time complexity = O(N).

With reverse dependency index:

    children = dep_index[p]

Time complexity = O(number_of_children).

For large corpora, this transforms dependency search from corpus-wide
scanning to local tree traversal.

This is essential for:

-   Object-of-verb searches
-   Subject extraction
-   Pattern statistics
-   Dependency-based collocation analysis

## Summary

The system uses:

-   Contiguous parallel token arrays
-   Sorted inverted indexes for fast lookup
-   Optional n-gram precomputation
-   Reverse dependency indexing for syntactic queries
-   Separate metadata storage for flexibility

Everything reduces to:

Integer arrays + sorted position lists + lightweight metadata.

This architecture scales linearly with corpus size and enables fast
structural and statistical search.
